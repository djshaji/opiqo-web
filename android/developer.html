Project Overview

Opiqo is a professional Guitar Multi-Effects Processor for Android that hosts and runs LV2 plugins in real time. The app provides a swipeable 4-pedal UI, full-duplex low-latency audio via Google Oboe, and support for 54 bundled guitar effects out of the box. The C++ DSP core (LiveEffectEngine, LV2Plugin, FileWriter) is shared verbatim with the Windows and Linux builds.

Core Capabilities

  • Floating-point precision audio processing
  • 32-bit WAV / FLAC lossless recording
  • Full LV2 plugin host via Lilv
  • Real-time audio I/O via Google Oboe
  • 4 independent LV2 plugin slots
  • LV2 buf-size specification compliant
  • Multi-ABI: arm64-v8a, armeabi-v7a, x86, x86_64

Recording Formats

  • WAV (32-bit, lossless)
  • FLAC (lossless)
  • MP3
  • Opus
  • OGG Vorbis

Key External Libraries

  • Google Oboe v1.10.0 тАФ low-latency audio
  • JACK (libjack) тАФ LV2 host utilities
  • Lilv тАФ LV2 plugin discovery and loading
  • Sord тАФ RDF triple store
  • Sratom тАФ Atom/RDF serialization
  • nlohmann/json тАФ preset serialization

Thread Ownership

  • UI thread тАФ all Java UI events, JNI calls to engine, plugin load/unload
  • Oboe audio thread тАФ capture, DSP process, render; no allocations, no Java calls
  • FileWriter thread тАФ encoding and disk write; runs in a background thread separate from the audio callback

Audio Signal Path

  • Oboe captures input samples on the audio thread
  • FullDuplexPass::onBothStreamsReady() receives input and output buffers
  • LiveEffectEngine::process() chains through slots 1тАУ4
  • Oboe writes output samples; FileWriter receives a copy if recording is active

LV2 buf-size Compliance

  • At stream start, the real Oboe burst size is queried
  • LV2Plugin is instantiated with bufsz:boundedBlockLength and correct min/max/nominal block lengths set from the actual burst size
  • This prevents plugins from assuming a fixed or arbitrary block size

Oboe Stream Configuration

  • Audio API: AAudio (preferred); falls back to OpenSL ES
  • Performance mode: LowLatency
  • Sharing mode: Exclusive (falls back to Shared)
  • Sample format: Float
  • Channel count: Stereo (from input, to output)

Real-time Safety Rules

  • No heap allocations in onBothStreamsReady()
  • No Java calls from the audio callback thread
  • No mutex locks in the hot path
  • Parameter updates from UI thread use the lock-free ring buffer
  • Atom messages queued in lv2_ringbuffer, consumed per cycle

LV2 Plugin Integration

Plugin Discovery

  • On first launch, all bundles under assets/lv2/ are copied to getFilesDir()/lv2/
  • Lilv's LilvWorld is pointed at the private files/lv2/ path
  • lilv_world_discover_all_bundles() loads all discovered plugins
  • No changes to Java or C++ code are needed when adding new bundles

Querying Port Information

LilvPlugin* plugin =
  lilv_plugins_get_by_uri(plugins, uri);

LILV_FOREACH(ports, i, ports) {
  LilvPort* port = lilv_ports_get(ports, i);

  bool is_audio  = lilv_port_is_a(
    plugin, port, node_AudioPort);
  bool is_control = lilv_port_is_a(
    plugin, port, node_ControlPort);
  bool is_atom   = lilv_port_is_a(
    plugin, port, node_AtomPort);
}

Querying Port Range (Sliders)

LilvNode *min_val = NULL,
         *max_val = NULL,
         *default_val = NULL;

lilv_port_get_range(
  plugin, port,
  &default_val, &min_val, &max_val);

float min = lilv_node_as_float(min_val);
float max = lilv_node_as_float(max_val);
float def = lilv_node_as_float(default_val);

// Clean up
lilv_node_free(min_val);
lilv_node_free(max_val);
lilv_node_free(default_val);

Atom Messaging for File Paths

Plugins that accept external file paths (IR loaders, cab simulators) use LV2 atom messages. The file path is written as an atom sequence into the lock-free ring buffer from the UI thread; the audio thread reads it during the next processing cycle. 

// 1. From the UI thread тАФ write atom sequence to ring buffer
LV2_Atom_Sequence* seq = (LV2_Atom_Sequence*) ringbuf_write_ptr;
seq->atom.type = uris->Sequence;
// ... write LV2_Atom_Event with file path body ...
ringbuffer.writeAdvance(total_size);

// 2. Audio thread тАФ read from ring buffer during processing
if (ringbuffer.readAvailable()) {
    LV2_Atom* atom = ringbuffer.readPtr();
    plugin->connectAtomPort(portIndex, atom);
    ringbuffer.readAdvance();
}

// 3. Plugin processes atom in onBothStreamsReady()
// State persists until a new atom message replaces it
Note: Send the file path atom once per file change. Some plugins hold the loaded file state in their internal buffers between audio cycles. Sending a null/empty atom will unload the file; resend the path to reload it.

JNI Bridge

All Java тЖФ native communication flows through jni_bridge.cpp. JNI methods are declared in AudioEngine.java and called from the UI thread only.

Engine Lifecycle

  • native boolean startEngine() тАФ opens Oboe streams with configured settings
  • native void stopEngine() тАФ closes streams, releases resources
  • native boolean isRunning() тАФ returns current audio engine state
  • native void setInputDevice(int deviceId) тАФ selects input endpoint
  • native void setOutputDevice(int deviceId) тАФ selects output endpoint

Plugin Management

  • native String[] getAvailablePlugins() тАФ returns list of plugin URIs from Lilv world
  • native boolean loadPlugin(int slot, String uri) тАФ loads plugin into slot 0тАУ3
  • native void unloadPlugin(int slot) тАФ unloads plugin and frees LV2 resources
  • native void setBypass(int slot, boolean bypass) тАФ enables/disables bypass for the slot

Parameter & Recording

  • native void setParameterValue(int slot, int portIndex, float value) тАФ queues parameter update via lock-free buffer
  • native void setMasterGain(float gain) тАФ sets output gain scaling
  • native void startRecording(int fd, int format, int quality) тАФ begins file recording
  • native void stopRecording() тАФ finalises and closes the output file
Important: The RECORD_AUDIO permission launcher must be registered in onCreate() тАФ not in response to a user interaction after the Activity is already in RESUMED state. If you encounter "LifecycleOwner is attempting to register while current state is RESUMED", move the launcher registration to MainActivity.onCreate().

UI Components

Main Screen Controls

  • Power Toggle тАФ master on/off; calls startEngine() / stopEngine(); requests RECORD_AUDIO if not already granted
  • Volume Slider тАФ calls setMasterGain(value)
  • Input Device Spinner тАФ populated from Android's AudioManager; calls setInputDevice(id)
  • Output Device Spinner тАФ calls setOutputDevice(id)
  • Record Button тАФ calls startRecording(fd, format, quality) / stopRecording()

Pedal Slot (ObjectFragment)

  • + Button тАФ opens plugin selection dialog backed by getAvailablePlugins()
  • Bypass Switch тАФ toggle in plugin header; calls setBypass(slot, state)
  • Delete Button тАФ calls unloadPlugin(slot), clears parameter panel
  • Parameter Sliders тАФ one per control port; generated by UI.java from LV2 port metadata; slider change calls setParameterValue(slot, portIndex, value)

Dynamic UI Generation (UI.java)

  • Queries LV2 control port count, labels, min, max, default via JNI after plugin load
  • Creates one SeekBar + label per port, scaled to [0, 100] and mapped to [min, max]
  • Atom-type ports (file path) are skipped in the slider UI; handled via atom messages separately if needed
  • All views are added to the slot's LinearLayout programmatically тАФ no layout XML changes needed for new plugins

Preset Management

Export

  • User taps Export in SettingsActivity
  • System document picker is opened with ACTION_CREATE_DOCUMENT
  • On file selection, getPresetList() is called via JNI тАФ returns all preset names and plugin configurations
  • Data is serialised to JSON with nlohmann/json and written to the chosen file descriptor

Import

  • User taps Import in SettingsActivity
  • System document picker opened with ACTION_OPEN_DOCUMENT
  • Selected JSON file is read; each preset object is passed to applyPreset() via JNI
  • Preset files are written to getFilesDir()/presets/ and the preset list is reloaded immediately тАФ no restart needed

Adding New LV2 Plugin Bundles

  1. 01

    Cross-Compile for Android ABIs

    Compile the plugin's .so library for each target ABI: arm64-v8a, armeabi-v7a, x86, x86_64 using the Android NDK toolchain. Ensure all dependencies are statically linked or provided as separate .so files.

  2. 02

    Create the LV2 Bundle Directory

    Create a directory under app/src/main/assets/lv2/ named <PluginName>.lv2/. Add the plugin's manifest.ttl and plugin-specific .ttl file(s) to this directory. Add the compiled .so library to app/src/main/jniLibs/<ABI>/.

  3. 03

    Rebuild тАФ No Code Changes Required

    Run ./gradlew assembleDebug. On next app launch, the assets copier runs and copies all bundles from assets/lv2/ to files/lv2/. Lilv discovers the new plugin automatically тАФ it will appear in the plugin selection dialog.

Note: The app copies bundles from assets/lv2/ to files/lv2/ only on the first launch (or after a fresh install). To force re-copy during development, clear the app's data in Android Settings or uninstall and reinstall.

Build System

Standard Android Gradle project using the Android Gradle Plugin with a CMake native build for the C++ code.

Common Gradle Tasks

# Debug APK
./gradlew assembleDebug

# Release APK (requires signing config)
