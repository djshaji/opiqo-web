§1

Objective

Build a production-ready Windows LV2 host that runs low-latency duplex audio with WASAPI, processes audio through LiveEffectEngine in real time, provides a Win32 desktop UI with 4 plugin slots in a 2×2 layout, and supports device selection, transport power toggle, and recording.

Primary Runtime Path

Audio input (WASAPI capture endpoint)
  → LiveEffectEngine::process(input, output, frames)
    → Audio output (WASAPI render endpoint)
§2

Scope and Non-Goals

In Scope v1.0

  • Win32 main window, dialogs, status bar, and control bar
  • WASAPI shared/exclusive duplex stream management
  • Input/output endpoint selection and hot-plug handling via IMMNotificationClient
  • Power toggle state machine: Off, Starting, Running, Stopping, Error
  • Plugin add/delete/enable/parameter UI mapped to LiveEffectEngine APIs
  • Recording controls with format (WAV/MP3/OGG) and quality selection
  • Settings dialog: sample rate, block size, share mode, endpoint pickers
  • Preset import/export via getPresetList/applyPreset

Out of Scope v1.0

  • Advanced theming and skinning
  • MIDI mapping and automation lanes
  • Sandboxed plugin process isolation
  • VST / AU / CLAP plugin formats
  • Multi-channel (5.1+) routing
  • Network/remote control features
§3

Architecture

Three cleanly separated layers with explicit thread ownership. The DSP layer is shared verbatim with the Linux and Android builds.

┌──────────────────────────────────────┐ │ Win32 UI Layer │ │ MainWindow, PluginSlot, ControlBar, │ │ PluginDialog, SettingsDialog │ │ Thread: UI/message thread │ └─────────────────┬────────────────────┘ │ calls (PostMessage for replies) ┌─────────────────▼────────────────────┐ │ Audio Platform Layer │ │ AudioEngine (WASAPI stream), │ │ WasapiDeviceEnum (IMMNotification) │ │ Thread: dedicated real-time thread │ └─────────────────┬────────────────────┘ │ calls (lock-free param queue) ┌─────────────────▼────────────────────┐ │ DSP Host Layer │ │ LiveEffectEngine + LV2Plugin │ │ (shared with Linux + Android) │ └──────────────────────────────────────┘

Layer Ownership

  • AudioEngine owns all COM/WASAPI resources and the real-time audio thread
  • LiveEffectEngine owns the plugin graph state and processing
  • UI layer owns user intents, persisted preferences, and state presentation
  • UI thread and audio thread communicate via preallocated lock-free buffers and posted window messages

State Machine

  • Off → Starting → Running (Power toggle on)
  • Running → Stopping → Off (Power toggle off)
  • Any → Error on WASAPI failure or device loss
  • Duplicate On/Off requests are ignored safely
  • Start failure keeps app in safe Off state

AudioEngine API

bool start(
  const std::string& inputDeviceId,
  const std::string& outputDeviceId,
  int sampleRate,
  int blockSize,
  ShareMode shareMode
);
void stop();

enum class State {
  Off, Starting, Running,
  Stopping, Error
};

State       state()        const;
std::string errorMessage() const;
uint32_t    dropoutCount() const;

Milestone Plan

11 milestones ordered to prove the real-time audio path first, then build the UI on top. Each milestone has concrete deliverables and acceptance criteria.

M0

Project Skeleton and Build Wiring

Create Windows-specific source structure and compile target.

  • Add source files under src/win32/ and src/main_win32.cpp
  • Update CMake target for MinGW cross-build with required Windows libs (ole32, uuid, mmdevapi)
  • Add resource files (resource.h, app.rc) and a minimal menu

Acceptance Criteria

  • App launches to a blank or minimal Win32 window
  • Clean compile in Debug; binary runs on Windows without missing DLL/runtime errors
M1

WASAPI Device Enumeration and Selection Model

Build stable endpoint discovery and selection storage.

  • WasapiDeviceEnum: implement capture/render listing; return DeviceInfo { id, friendlyName, isDefault }; implement IMMNotificationClient change notifications
  • AppSettings: fields inputDeviceId, outputDeviceId, sampleRate, blockSize, shareMode, recordFormat, recordQuality
  • Implement load/save at startup/shutdown to %APPDATA%\Opiqo\settings.json

Acceptance Criteria

  • Device hot-plug refreshes list without requiring app restart
  • Missing saved device falls back to default system endpoint
M2

AudioEngine Core (Pass-Through, No DSP)

Prove full-duplex WASAPI pipeline and stream lifecycle.

  • Implement AudioEngine::start() / stop() with sampleRate, blockSize, deviceIds, shareMode
  • Initialization sequence: CoInitializeEx, device activation, format negotiation, event-callback mode
  • Dedicated real-time audio thread — no allocations in the loop; MMCSS priority set via AvSetMmThreadCharacteristics
  • Basic pass-through path without LiveEffectEngine processing

Acceptance Criteria

  • 15-minute pass-through run without crash or dropout
  • Stop/start repeated 50 times without leaked COM handles
M3

LiveEffectEngine Integration in Audio Thread

Insert DSP host into capture → render loop.

  • Replace pass-through with LiveEffectEngine::process(input, output, frames)
  • Implement buffer conversion path supporting float and fallback format conversion from WASAPI PCM
  • Verify stereo channel handling for the process path
  • Validate bypass: with no plugins loaded, signal passes through cleanly

Acceptance Criteria

  • With no plugins loaded: audible clean pass-through
  • With plugins loaded: audible effect processing in slot order
  • No allocations in the hot path (verified with Application Verifier)
M4

Power Toggle State Machine and Transport Control

Make the power toggle the single source of truth for engine transport state.

  • Implement atomic engine state: Off, Starting, Running, Stopping, Error
  • Wire ControlBar power toggle: On → AudioEngine::start() with selected endpoints; Off → stop()
  • Keep toggle On only after successful start; revert to Off on failure
  • Error path surfaces message in status bar; keeps app in safe state

Acceptance Criteria

  • Duplicate On/Off requests are ignored safely
  • Error path transitions to Error and exposes message in status bar
M5

MainWindow Layout and 2×2 Plugin Slot Grid

Implement the required desktop layout with proper resizing.

  • Create main frame with menu bar and status bar
  • Implement 2×2 slot region: top row (Slot 1, Slot 2); bottom row (Slot 3, Slot 4)
  • Add ControlBar: Power toggle, Gain slider, Record button, format dropdown, device selectors
  • Handle WM_SIZE to keep slots proportional; enforce minimum size 900 × 650

Acceptance Criteria

  • Layout remains usable and proportional at minimum size 900 × 650
  • Resizing preserves slot proportions and control visibility
M6

Plugin Management UI

Connect plugin lifecycle controls to existing LiveEffectEngine APIs.

  • PluginDialog: data source from getAvailablePlugins(); list view with name filter
  • Per-slot controls: Add plugin, Bypass/Enable toggle, Delete plugin
  • Reflect active plugin name and bypass status in the slot header label
  • All slot operations are independent; UI remains responsive during plugin load

Acceptance Criteria

  • Each of the 4 slots independently manages its plugin instance
  • UI remains responsive during plugin add/delete operations
M7

Dynamic Parameter Panels

Build runtime-generated Win32 controls from plugin port metadata.

  • Generate controls per port type: Float (range) → trackbar/slider; Toggle → checkbox; Enumeration → combobox; Atom file path → Browse button + GetOpenFileName
  • Implement value mapping: min/max/default clamp; on-change calls engine_->setValue(slot, portIdx, value)
  • Ensure real-time-safe parameter handoff — enqueue to lock-free buffer consumed in audio thread

Acceptance Criteria

  • Control movements update sound in real time without dropouts
  • No audible glitches from rapid slider movements
M8

Recording UX and Format Handling

Complete recording workflow from the UI.

  • Record button: on → engine_->startRecording(fd, format, quality); off → stopRecording()
  • Bind format dropdown: 0=WAV, 1=MP3, 2=OGG; show/hide quality selector for lossy formats only
  • Connect selected format/quality to startRecording arguments
  • Output file saved to %USERPROFILE%\Music\ with timestamp filename

Acceptance Criteria

  • Recorded files open successfully in external players without corruption
  • Record can run while all 4 plugin slots are actively processing
  • File is finalized cleanly even if the app exits during recording
M9

Settings Dialog and Runtime Reconfiguration

Enable configurable audio preferences and preset import/export.

  • Audio settings tab: sample rate, block size, share mode; explicit input/output endpoint pickers populated by WasapiDeviceEnum
  • Apply flow: if engine is running → stop → apply settings → restart with new values
  • Presets tab: Export → GetSaveFileName dialog → writes getPresetList() to JSON; Import → GetOpenFileName → reads JSON → calls applyPreset() per slot
  • Settings saved via AppSettings::save() on dialog OK

Acceptance Criteria

  • Settings survive app restart
  • Unsupported setting combinations fail gracefully with a fallback
  • Import of a preset restores all four slot configurations
M10

Hardening, Performance, and Release Readiness

Production stability and diagnosability.

  • Add rate-limited error logging to %APPDATA%\Opiqo\opiqo.log using logging_macros.h
  • Add MMCSS priority setup for the audio thread via AvSetMmThreadCharacteristics
  • Device hot-plug stress test: unplug/replug USB audio device 20× while running
  • Verify no blocking operations in the audio loop (no mutex, no malloc, no Win32 waits)
  • Dropout counter displayed in the status bar

Acceptance Criteria

  • 60-minute continuous run without dropout under normal load
  • Device loss/recovery path works without app restart
  • Zero memory leaks on clean startup/shutdown (verified with Application Verifier or Dr. Memory)
§5

Cross-cutting Technical Requirements

Real-time Safety (Audio Loop)

  • No heap allocations inside the WASAPI callback loop
  • No blocking UI locks inside the audio loop
  • Preallocate all conversion and transfer buffers in AudioEngine::start()
  • Lock-free queue for plugin parameter updates from UI thread

Threading

  • UI thread handles all window messages and user actions
  • AudioEngine thread handles capture/process/render only
  • Long operations (plugin load, file I/O) must not block the UI thread
  • PostMessage used to marshal audio events back to the UI thread

Error Handling

  • Surface all critical WASAPI failures in the status bar
  • On start failure, keep app in safe Off state with error message
  • Handle invalidated/lost devices with a recoverable workflow
  • AppSettings::load() returns safe defaults on any file error

COM Initialization

  • CoInitializeEx(nullptr, COINIT_MULTITHREADED) on the audio thread
  • CoInitializeEx(nullptr, COINIT_APARTMENTTHREADED) on the UI thread
  • All WASAPI interfaces released before CoUninitialize
  • Smart pointer wrappers (ComPtr / CComPtr) to prevent leaks

Test Plan

Functional Tests

  • App startup with valid and missing saved devices
  • Power toggle on/off cycles (50 repetitions)
  • Input/output device switch while engine is running
  • Add/delete plugins in all 4 slots
  • Parameter changes under active audio
  • Recording for each supported format (WAV, MP3, OGG)

Robustness Tests

  • USB headset unplug/replug while engine is running
  • Rapid power toggle spam (100×)
  • Open Settings and Apply repeatedly during idle and running states
  • Shared mode ↔ exclusive mode switch at runtime

Performance Tests

  • CPU usage: no plugins, 1 plugin, 4 plugins
  • Dropout/glitch count under stress: 10 minutes at 48 kHz / 256-sample buffer
  • Latency: shared vs exclusive mode comparison
  • 60-minute soak run (release readiness)
  • Application Verifier / Dr. Memory for leak detection
§8

Build System

Windows builds use the windows-default CMake preset. Cross-compilation from Linux is supported via MinGW-w64.

Native Windows Build (MSVC)

git clone https://github.com/djshaji/opiqo-windows.git
cd opiqo-windows

# Configure
cmake --preset windows-default

# Build
cmake --build build-windows --config Release -j4

Cross-compile from Linux (MinGW)

sudo apt install mingw-w64

git clone https://github.com/djshaji/opiqo-windows.git
cd opiqo-windows

cmake --preset windows-cross
cmake --build build-windows-cross -j$(nproc)

CMakePresets.json — windows preset

{
  "name": "windows-default",
  "displayName": "Windows (Win32 + WASAPI)",
  "binaryDir": "${sourceDir}/build-windows",
  "cacheVariables": {
    "CMAKE_BUILD_TYPE": "Release",
    "OPIQO_TARGET_PLATFORM": "windows"
  }
}
§9

Sequencing Recommendation

This order minimises risk by proving the real-time audio path early before investing heavily in UI complexity.

  1. M0–M3 — Project skeleton → WASAPI device enum → AudioEngine pass-through → DSP integration. Audio foundation first.
  2. M4–M6 — Transport state machine → shell layout → plugin management. Core UI on proven audio.
  3. M7–M9 — Dynamic parameter panels → recording UX → settings dialog. Full feature surface.
  4. M10 — Hardening, MMCSS priority, logging, soak tests. Release readiness.
